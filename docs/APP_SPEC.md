# APP_SPEC.md — GrowDay

เอกสารนี้คือ **สเปกระบบหลัก (Application Specification)** สำหรับเว็บแอพ **GrowDay** ใช้กำกับการพัฒนาให้ตรงตามเป้าหมายของรุ่นแรก (MVP) และใช้เป็นแหล่งอ้างอิงหลักสำหรับทีมพัฒนา, Codex, AI coding agent หรือผู้ที่ต้องการปรับปรุงระบบในอนาคต

> หากมีความขัดแย้งระหว่างเอกสารนี้กับไฟล์อื่น ให้ยึด `APP_SPEC.md` เป็นหลักในเรื่อง “ขอบเขตระบบและฟังก์ชันที่ต้องมี/ไม่ต้องมี”  
> รายละเอียดเชิงเทคนิคของฐานข้อมูลและ API ให้แยกไปอ้างอิงใน `DATA_SCHEMA.md` และ `API_SPEC.md` เมื่อมีการสร้างไฟล์เหล่านั้นเพิ่มเติม

## 2. วัตถุประสงค์ของระบบ

GrowDay ถูกออกแบบขึ้นเพื่อช่วยให้กลุ่มผู้ปลูกผักไฮโดรโปนิกส์สามารถบันทึกและติดตามรอบการปลูกได้ง่ายขึ้น โดยเฉพาะการรู้ว่า “ปลูกอะไร เมื่อไร จะเก็บเกี่ยวได้วันไหน และได้ผลผลิตเท่าไร”

### 2.1 วัตถุประสงค์หลัก

1. ให้สมาชิกกลุ่มบันทึกข้อมูลการปลูกผักแต่ละรอบได้ง่าย
2. ให้ระบบคำนวณวันที่คาดว่าจะเก็บเกี่ยวได้อัตโนมัติจากชนิดผักและวันที่ปลูก
3. แสดงรายการผักที่กำลังปลูก พร้อมจำนวนวันที่เหลือก่อนเก็บเกี่ยว
4. ช่วยให้รู้ล่วงหน้าว่าผักใดใกล้ถึงรอบเก็บเกี่ยว
5. บันทึกผลผลิตจริงหลังเก็บเกี่ยวได้
6. สรุปภาพรวมการปลูกและผลผลิตเบื้องต้นได้
7. ใช้งานบนมือถือได้สะดวก ไม่ซับซ้อน
8. ใช้เครื่องมือฟรีหรือต้นทุนต่ำ เหมาะกับงานชุมชน

### 2.2 ปัญหาที่ระบบต้องช่วยแก้

| ปัญหาเดิม | สิ่งที่ GrowDay ต้องช่วยแก้ |
|---|---|
| จำวันปลูกและวันเก็บเกี่ยวด้วยการจดมือ | บันทึกในเว็บแอพและคำนวณวันเก็บเกี่ยวอัตโนมัติ |
| ไม่รู้ว่ารอบไหนใกล้เก็บเกี่ยว | Dashboard แสดงรายการใกล้ถึงวันเก็บเกี่ยว |
| ผลผลิตแต่ละรอบกระจัดกระจาย | บันทึกผลผลิตจริงลง Google Sheets |
| ใช้เอกสารหรือสมุดจดแยกหลายชุด | รวมข้อมูลไว้ในระบบเดียว |
| สมาชิกบางคนไม่ถนัดระบบซับซ้อน | ออกแบบให้กดง่าย ใช้เมนูน้อย และไม่ต้องล็อกอิน |

## 3. กลุ่มผู้ใช้และบริบทการใช้งาน

### 3.1 กลุ่มผู้ใช้หลัก

- สมาชิกกลุ่มต่อยอดถุงทองผักสด
- ผู้รับผิดชอบการปลูกผักไฮโดรโปนิกส์
- ผู้ดูแลการเก็บเกี่ยวและบันทึกผลผลิต
- ผู้ประสานงานหรือผู้สรุปผลผลิตของกลุ่ม

### 3.2 พฤติกรรมการใช้งานที่คาดหวัง

ผู้ใช้เปิดเว็บแอพผ่านมือถือ แล้วใช้งานตามขั้นตอนง่าย ๆ ดังนี้

1. เปิดหน้าเว็บแอพ
2. กด “บันทึกการปลูก”
3. เลือกชนิดผัก
4. ใส่จำนวนช่องปลูกหรือจำนวนต้น
5. เลือกวันที่ปลูก
6. ระบบแสดงวันที่คาดว่าจะเก็บเกี่ยวได้
7. กดบันทึก
8. เมื่อถึงวันเก็บเกี่ยว ให้กด “บันทึกเก็บเกี่ยว”
9. ใส่ผลผลิตจริง เช่น 12.5 กก.
10. ดูสรุปผลผลิตย้อนหลังเบื้องต้น

## 4. แนวคิดสำคัญของระบบ

### 4.1 ความหมายของ “วันที่ปลูก”

ในระบบ GrowDay ให้กำหนดนิยามที่ชัดเจนว่า:

> **วันที่ปลูก = วันที่ย้ายต้นกล้าลงราง/ช่องปลูกไฮโดรโปนิกส์จริง**

ไม่ใช่วันที่เพาะเมล็ด

เหตุผล:

- ระยะเพาะกล้าอาจไม่เท่ากันในแต่ละรอบ
- ผู้ใช้เข้าใจง่ายกว่า
- คำนวณวันเก็บเกี่ยวได้สม่ำเสมอกว่า
- ลดความซับซ้อนของระบบ MVP

### 4.2 หน่วยการปลูกหลัก

หน่วยหลักที่ระบบควรใช้คือ:

> **จำนวนช่องปลูก**

เหตุผล:

- เหมาะกับระบบไฮโดรโปนิกส์ที่มีราง/แผง/โต๊ะปลูก
- โดยทั่วไป 1 ช่องปลูกสัมพันธ์กับ 1 ต้น
- ใช้คำนวณและประเมินผลผลิตได้ง่าย
- ผู้ใช้ในกลุ่มเข้าใจได้ง่ายกว่าการกรอกพื้นที่เป็นตารางเมตร

### 4.3 หน่วยที่ระบบควรรองรับ

| หน่วย | สถานะ | หมายเหตุ |
|---|---|---|
| จำนวนช่องปลูก | ค่าเริ่มต้น | ใช้เป็นหลักใน MVP |
| จำนวนต้น | รองรับ | ใช้กรณีนับเป็นต้น |
| ราง/โต๊ะปลูก | ยังไม่ใช้คำนวณหลัก | อาจใส่ในหมายเหตุหรือเวอร์ชันต่อยอด |

## 6. ขอบเขตระบบรุ่นแรก (MVP Scope)

MVP ของ GrowDay ต้องเป็นระบบที่เล็กพอให้สร้างได้จริง แต่ครบพอสำหรับใช้งานในกลุ่มได้ทันที

### 6.1 ฟังก์ชันที่ต้องมีใน MVP

| รหัส | ฟังก์ชัน | รายละเอียด |
|---|---|---|
| MVP-01 | หน้าแรก / Dashboard | แสดงภาพรวมการปลูกและรายการใกล้เก็บเกี่ยว |
| MVP-02 | บันทึกการปลูก | เลือกผัก ใส่จำนวน ใส่วันที่ปลูก และบันทึกข้อมูล |
| MVP-03 | คำนวณวันเก็บเกี่ยวอัตโนมัติ | ใช้วันที่ปลูก + อายุเก็บเกี่ยวของผัก |
| MVP-04 | รายการปลูกทั้งหมด | แสดงรายการที่เคยบันทึกไว้ทั้งหมด |
| MVP-05 | ค้นหา/กรองรายการปลูก | กรองตามชื่อผักและสถานะ |
| MVP-06 | แก้ไขรายการปลูก | แก้ไขข้อมูลที่บันทึกผิดได้ |
| MVP-07 | ลบ/ยกเลิกรายการปลูก | ลบหรือยกเลิกรายการที่ไม่ต้องการใช้ |
| MVP-08 | บันทึกเก็บเกี่ยว | บันทึกวันที่เก็บเกี่ยวจริงและผลผลิตจริง |
| MVP-09 | สรุปผลผลิตเบื้องต้น | แสดงผลผลิตรวมรายเดือนและรายการล่าสุด |
| MVP-10 | เชื่อม Google Sheets | บันทึกและดึงข้อมูลผ่าน Google Apps Script |

### 6.2 ฟังก์ชันที่ยังไม่ต้องทำใน MVP

รายการต่อไปนี้ **ยังไม่ต้องทำในรุ่นแรก** เพื่อไม่ให้ระบบใหญ่เกินไป

1. ระบบล็อกอิน/สมัครสมาชิก
2. ระบบแยกสิทธิ์ผู้ใช้หลายระดับ
3. ระบบแจ้งเตือนผ่าน LINE / Email / Push Notification
4. ระบบสต็อกเมล็ดพันธุ์ ปุ๋ย อุปกรณ์
5. ระบบต้นทุน/กำไรละเอียด
6. ระบบขายสินค้า/รับออเดอร์
7. ระบบแผนผังโต๊ะปลูกแบบลากวาง
8. ระบบแนบรูปภาพแต่ละรอบปลูก
9. ระบบ Offline-first เต็มรูปแบบ
10. ระบบหลายภาษา
11. ระบบ Dashboard ขั้นสูงพร้อมกราฟซับซ้อน
12. ระบบสมาชิกหลายกลุ่ม/หลายฟาร์ม

> หลักการสำคัญ: รุ่นแรกต้องเน้น “บันทึกปลูก → คำนวณเก็บเกี่ยว → บันทึกผลผลิต → ดูสรุป” ให้ใช้งานได้จริงก่อน

## 7. โครงสร้างหน้าจอที่ต้องมี

ระบบออกแบบเป็น Mobile-first โดยมีหน้าหลัก 5 หน้า

| หน้า | ชื่อไฟล์แนะนำ | วัตถุประสงค์ |

## 8. รายละเอียดหน้าจอ

## 8.1 หน้าแรก / Dashboard

### เป้าหมาย

เปิดระบบมาแล้วเห็นสถานการณ์ภาพรวมทันที

### สิ่งที่ต้องแสดง

1. ชื่อระบบ `GrowDay`
2. คำอธิบายสั้น ๆ เช่น `จัดการการปลูกผักไฮโดรโปนิกส์`
3. การ์ดสรุป 4 รายการ
   - กำลังปลูก
   - เก็บเกี่ยววันนี้
   - เก็บเกี่ยวภายใน 7 วัน
   - ผลผลิตเดือนนี้
4. เมนูด่วน
   - บันทึกการปลูก
   - รายการปลูก
   - บันทึกเก็บเกี่ยว
   - สรุปผลผลิต
5. รายการใกล้ถึงวันเก็บเกี่ยว
6. รายการปลูกล่าสุด

### ตัวอย่างข้อมูล

| รายการ | ตัวอย่าง |
|---|---:|
| กำลังปลูก | 24 รายการ |
| เก็บเกี่ยววันนี้ | 3 รายการ |
| เก็บเกี่ยวภายใน 7 วัน | 9 รายการ |
| ผลผลิตเดือนนี้ | 186 กก. |

## 8.2 หน้าบันทึกการปลูก

### เป้าหมาย

ให้ผู้ใช้บันทึกข้อมูลรอบปลูกใหม่ได้รวดเร็ว และเห็นวันที่คาดว่าเก็บเกี่ยวได้ทันที

### ช่องข้อมูล

| ช่องข้อมูล | ประเภท | จำเป็น | ค่าเริ่มต้น/ตัวอย่าง |
|---|---|---|---|
| ชนิดผัก | Dropdown | ใช่ | Green Cos |
| หน่วยปลูก | Dropdown | ใช่ | จำนวนช่องปลูก |
| จำนวนที่ปลูก | Number input | ใช่ | 120 |
| วันที่ปลูก | Date input | ใช่ | 2026-06-05 |
| วันที่คาดว่าเก็บเกี่ยวได้ | Auto result | ใช่ | 2026-06-30 |
| หมายเหตุ | Textarea | ไม่จำเป็น | แหล่งต้นกล้า, สูตรปุ๋ย, รางปลูก |

### ปุ่ม

- `บันทึกการปลูก`
- `ล้างข้อมูล`

### เงื่อนไขการทำงาน

1. เมื่อเลือกชนิดผัก ระบบต้องรู้ `harvestAgeDays` ของผักนั้น
2. เมื่อเลือกวันที่ปลูก ระบบต้องคำนวณวันที่เก็บเกี่ยวอัตโนมัติ
3. หากยังไม่ได้เลือกผักหรือวันที่ปลูก ไม่ควรให้บันทึก
4. จำนวนที่ปลูกต้องมากกว่า 0
5. หลังบันทึกสำเร็จ ให้แสดงข้อความสำเร็จและล้างฟอร์มหรือกลับไปหน้ารายการปลูก

### เป้าหมาย

ให้ผู้ใช้เห็นรายการปลูกทั้งหมด และจัดการข้อมูลได้ง่ายบนมือถือ

### สิ่งที่ต้องมี

1. ช่องค้นหาชื่อผัก
2. ตัวกรองสถานะ
   - ทั้งหมด
   - กำลังปลูก
   - พร้อมเก็บ
   - เก็บแล้ว
3. การ์ดสรุปจำนวนรายการ
4. รายการปลูกในรูปแบบการ์ด ไม่จำเป็นต้องเป็นตารางแนวนอน

### ข้อมูลที่ต้องแสดงในแต่ละรายการ

| ข้อมูล | ตัวอย่าง |
|---|---|
| ชื่อผัก | Green Cos |
| วันที่ปลูก | 05/06/2026 |
| วันที่คาดว่าเก็บเกี่ยวได้ | 30/06/2026 |
| อีกกี่วันจะถึงวันเก็บเกี่ยว | เหลือ 2 วัน |
| สถานะ | กำลังปลูก / พร้อมเก็บ / เก็บแล้ว |
| จำนวนที่ปลูก | 120 ช่อง |
| ผลผลิตจริง | 12.5 กก. หรือ `-` |

### ปุ่มการทำงานในแต่ละรายการ

- `แก้ไข`
- `เก็บเกี่ยว`
- `ลบ` หรือ `ยกเลิก`

> บนมือถือให้ใช้ Card List แทน Table จริง เพราะอ่านง่ายกว่าและไม่ล้นจอ

## 8.4 หน้าบันทึกเก็บเกี่ยว

### เป้าหมาย

ให้ผู้ใช้เลือกแปลง/รายการปลูกที่ถึงรอบเก็บเกี่ยว แล้วบันทึกผลผลิตจริงได้

### ช่องข้อมูล

| ช่องข้อมูล | ประเภท | จำเป็น | หมายเหตุ |
|---|---|---|---|
| รายการปลูก | Dropdown | ใช่ | แสดงเฉพาะรายการที่ยังไม่เก็บเกี่ยว |
| วันที่ปลูก | Read only | ใช่ | ดึงจากข้อมูลเดิม |
| วันที่คาดว่าเก็บเกี่ยว | Read only | ใช่ | ดึงจากข้อมูลเดิม |
| จำนวนที่ปลูก | Read only | ใช่ | ดึงจากข้อมูลเดิม |
| วันที่เก็บเกี่ยวจริง | Date input | ใช่ | ค่าเริ่มต้นเป็นวันนี้ |
| ปริมาณผลผลิต | Number input | ใช่ | เช่น 12.5 |
| หน่วยผลผลิต | Dropdown | ใช่ | กก. |
| หมายเหตุ | Textarea | ไม่จำเป็น | คุณภาพ, ลูกค้า, ปัญหา |

### เงื่อนไขการทำงาน

1. ต้องเลือกข้อมูลรอบปลูกก่อนบันทึก
2. ปริมาณผลผลิตต้องมากกว่า 0
3. เมื่อบันทึกแล้ว สถานะรายการต้องเปลี่ยนเป็น `harvested`
4. ต้องเก็บวันที่เก็บเกี่ยวจริงและผลผลิตจริงไว้ใน Google Sheets

## 9. สถานะของรายการปลูก

ระบบควรใช้สถานะมาตรฐานดังนี้

| สถานะในระบบ | ความหมาย | เงื่อนไข |
|---|---|---|
| `growing` | กำลังปลูก | ยังไม่ถึงวันเก็บเกี่ยว และยังไม่บันทึกผลผลิต |
| `ready` | พร้อมเก็บ | ถึงหรือเลยวันที่คาดว่าเก็บเกี่ยวแล้ว แต่ยังไม่บันทึกผลผลิต |
| `harvested` | เก็บแล้ว | บันทึกผลผลิตจริงแล้ว |
| `cancelled` | ยกเลิก | รายการถูกยกเลิก ไม่ใช้คำนวณผลผลิต |

### 9.1 Logic การประเมินสถานะ

ระบบสามารถประเมินสถานะอัตโนมัติจากวันที่ได้

```js
if (status === 'harvested') {
  displayStatus = 'เก็บแล้ว'
} else if (today >= harvestDate) {
  displayStatus = 'พร้อมเก็บ'
} else {
  displayStatus = 'กำลังปลูก'
}
```

## 10. Logic การคำนวณหลัก

### 10.1 คำนวณวันที่คาดว่าเก็บเกี่ยวได้

```js
harvestDate = plantedDate + harvestAgeDays
```

ตัวอย่าง:

| รายการ | ค่า |
|---|---|
| ชนิดผัก | Green Cos |
| วันที่ปลูก | 2026-06-05 |
| อายุเก็บเกี่ยว | 25 วัน |
| วันที่คาดว่าเก็บเกี่ยว | 2026-06-30 |

### 10.2 คำนวณจำนวนวันที่เหลือ

```js
daysRemaining = harvestDate - today
```

การแสดงผล:

| ค่า | แสดงผล |
|---:|---|
| มากกว่า 0 | เหลือ X วัน |
| เท่ากับ 0 | เก็บเกี่ยววันนี้ |
| น้อยกว่า 0 | เลยกำหนด X วัน |
| บันทึกเก็บเกี่ยวแล้ว | เก็บแล้ว |

### 10.3 คำนวณผลผลิตคาดการณ์เบื้องต้น

ใน MVP สามารถมีหรือไม่มีก็ได้ แต่ถ้ามี ให้ใช้สูตรง่าย ๆ

```js
estimatedYield = quantity * averageWeightKg
```

ตัวอย่าง:

```txt
จำนวนช่องปลูก 120 ช่อง
น้ำหนักเฉลี่ย Green Cos 0.12 กก./ต้น
ผลผลิตคาดการณ์ = 120 x 0.12 = 14.4 กก.
```

> ค่านี้เป็นเพียงการประมาณ ไม่ใช่ผลผลิตจริง  
> ผลผลิตจริงต้องบันทึกจากหน้าบันทึกเก็บเกี่ยว

## 20. Roadmap หลัง MVP

หลังจาก MVP ใช้งานได้จริงแล้ว สามารถพัฒนาต่อได้ตามลำดับนี้

### Phase 2

1. เพิ่มหน้าตั้งค่าชนิดผัก

4. เพิ่ม Export CSV
5. เพิ่มการกรองตามเดือน/ปี

2. เพิ่มระบบต้นทุนเบื้องต้น
3. เพิ่มแนบรูปภาพผลผลิต
4. เพิ่มระบบสต็อกเมล็ดพันธุ์/ปุ๋ย

สร้างเว็บแอพ GrowDay ตามเอกสาร docs/APP_SPEC.md โดยยึดขอบเขต MVP เท่านั้น

## 22. สรุปขอบเขตสุดท้าย

GrowDay รุ่น MVP ต้องเป็นเว็บแอพที่ทำ 4 เรื่องให้ดีและใช้งานจริงได้ก่อน

1. **บันทึกการปลูก**
2. **คำนวณวันเก็บเกี่ยว**
3. **ติดตามรายการปลูก**
4. **บันทึกผลผลิตจริง**

สิ่งสำคัญที่สุดคือระบบต้องเรียบง่าย ใช้งานง่ายบนมือถือ และไม่หลุดไปเป็นระบบฟาร์มขนาดใหญ่ที่ซับซ้อนเกินความจำเป็น
