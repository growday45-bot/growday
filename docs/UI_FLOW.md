# UI_FLOW.md

เอกสารกำกับหน้าจอและลำดับการใช้งานเว็บแอพ **GrowDay**  
ระบบจัดการการปลูกผักไฮโดรโปนิกส์สำหรับ **กลุ่มต่อยอดถุงทองผักสด**

> เอกสารนี้ใช้ควบคุมการออกแบบ UI/UX และลำดับการทำงานของเว็บแอพ ให้ตรงกับรุ่นแรก (MVP) และสามารถสร้างจริงได้ด้วย HTML5 + Tailwind CSS CDN + Vanilla JS + Google Apps Script + Google Sheets

---

## 1. วัตถุประสงค์ของเอกสารนี้

`UI_FLOW.md` ใช้เป็นเอกสารอ้างอิงสำหรับ

- กำหนดหน้าจอที่ต้องมีในเว็บแอพ GrowDay รุ่น MVP
- กำหนดลำดับการใช้งานของผู้ใช้ตั้งแต่เปิดแอพจนถึงบันทึกข้อมูล
- ควบคุมให้ UI ทำตามภาพตัวอย่างที่ออกแบบไว้ ไม่เพิ่มความซับซ้อนเกินจำเป็น
- ช่วยให้ Codex / AI coding agent / ทีมพัฒนาเข้าใจว่าแต่ละหน้าควรแสดงอะไร
- ลดปัญหาการสร้างหน้าจอเกินสเปก หรือสร้างระบบที่สวยแต่ใช้งานจริงยาก

---

## 2. หลักการออกแบบ UI โดยรวม

เว็บแอพนี้เป็นเครื่องมือใช้งานจริงในกลุ่มผู้ปลูกผัก ไม่ใช่เว็บประชาสัมพันธ์ ดังนั้น UI ต้องเน้นความง่าย ชัดเจน และใช้งานบนมือถือได้ดี

### 2.1 แนวทางภาพรวม

- Mobile-first รองรับมือถือเป็นหลัก
- ใช้หน้าจอแนวตั้งเป็นหลัก
- ใช้ภาษาไทยเป็นหลัก
- ใช้ชื่อผักภาษาอังกฤษตามชื่อที่กลุ่มใช้งาน เช่น Green Cos, Red Oak
- ไม่ต้องมีระบบ Login
- เปิดเว็บแล้วเข้าใช้งานได้ทันที
- ปุ่มต้องใหญ่พอสำหรับการแตะบนมือถือ
- ข้อมูลสำคัญต้องอ่านง่าย ไม่แน่นเกินไป
- ใช้โครงสร้างที่สร้างได้จริงด้วย HTML + Tailwind + Vanilla JS

### 2.2 โทนสี

| รายการ | แนวทาง |
|---|---|
| สีหลัก | เขียว |
| สีพื้นหลัง | ขาว / เทาอ่อนมาก |
| สีการ์ด | ขาว |
| สีตัวอักษรหลัก | ดำ / เทาเข้ม |
| สีตัวอักษรรอง | เทา |
| สีแจ้งเตือนใกล้เก็บเกี่ยว | เขียวอ่อน / เหลืองอ่อน |
| สีสถานะเก็บแล้ว | เทาอ่อน |

### 2.3 รูปแบบองค์ประกอบ

- ใช้ card layout เป็นหลัก
- ขอบมน `rounded-xl` หรือ `rounded-2xl`
- เงาอ่อน ๆ หรือเส้นขอบสีเทาอ่อน
- ไม่ใช้ animation ซับซ้อน
- ไม่ใช้ glassmorphism หนัก ๆ
- ไม่ใช้ layout ที่ต้องใช้ library เพิ่มเติม
- ไม่ใช้กราฟซับซ้อนใน MVP

## 3. โครงสร้างหน้าจอหลักของ MVP

รุ่นแรก (MVP) มี 4 หน้า/แท็บหลัก

```txt
1. หน้าแรก / Dashboard
2. บันทึกการปลูก
3. รายการปลูก
4. บันทึกเก็บเกี่ยว
```

การเข้าถึงหน้าต่าง ๆ ใช้ Bottom Navigation 4 เมนู

```txt
หน้าแรก | ปลูก | เก็บเกี่ยว | สรุป
```

หมายเหตุ: เมนู `สรุป` ใน MVP สามารถเริ่มจากหน้าสรุปแบบเรียบง่าย หรือแสดงเฉพาะสรุปผลผลิตพื้นฐานก่อน หากยังไม่ทำหน้าสรุปเต็ม ให้ใช้ข้อมูลสรุปในหน้าแรกแทนได้

## 4. โครงสร้าง Navigation

### 4.1 Bottom Navigation

แสดงอยู่ด้านล่างของหน้าจอทุกหน้า

| เมนู | หน้าที่เชื่อมโยง | สถานะ |
|---|---|---|
| หน้าแรก | Dashboard | ต้องมี |
| ปลูก | บันทึกการปลูก / รายการปลูก | ต้องมี |
| เก็บเกี่ยว | บันทึกเก็บเกี่ยว | ต้องมี |
| สรุป | สรุปผลผลิต | ทำแบบพื้นฐานใน MVP ได้ |

### 4.2 พฤติกรรมของ Bottom Navigation

- เมนูที่กำลังใช้งานต้องเป็นสีเขียว
- เมนูอื่นเป็นสีเทา
- กดแล้วเปลี่ยน view โดยไม่จำเป็นต้องเปลี่ยนหน้าใหม่จริงก็ได้
- สามารถทำเป็น single-page app แบบง่ายด้วย Vanilla JS ได้
- ไม่ต้องใช้ router library

ตัวอย่างแนวทาง implementation:

```txt
ใช้ data-view หรือ id ของ section แล้วสลับ display/block hidden
```

## 6. หน้าแรก / Dashboard

### 6.1 วัตถุประสงค์ของหน้า

หน้าแรกใช้ให้ผู้ใช้เห็นภาพรวมทันทีว่า

- ตอนนี้มีรายการปลูกกี่รายการ
- วันนี้มีรายการที่ควรเก็บเกี่ยวกี่รายการ
- ภายใน 7 วันจะมีรายการเก็บเกี่ยวกี่รายการ
- เดือนนี้ได้ผลผลิตรวมเท่าไหร่
- รายการไหนใกล้ถึงวันเก็บเกี่ยว
- รายการปลูกล่าสุดคืออะไร

### 6.2 องค์ประกอบที่ต้องมี

#### 6.2.1 Header

```txt
GrowDay
จัดการการปลูกผักไฮโดรโปนิกส์
```

#### 6.2.2 Summary Cards

แสดงการ์ดสรุป 4 ช่อง

| การ์ด | ข้อมูลที่แสดง |
|---|---|
| กำลังปลูก | จำนวนรายการที่ status = growing หรือ ready |
| เก็บเกี่ยววันนี้ | จำนวนรายการที่ harvestDate = วันนี้ และยังไม่ harvested |
| เก็บเกี่ยวภายใน 7 วัน | จำนวนรายการที่ harvestDate อยู่ในช่วงวันนี้ถึงอีก 7 วัน |
| ผลผลิตเดือนนี้ | ผลรวม actualYield ของเดือนปัจจุบัน |

ตัวอย่างข้อมูล:

```txt
กำลังปลูก 24 รายการ
เก็บเกี่ยววันนี้ 3 รายการ
เก็บเกี่ยวภายใน 7 วัน 9 รายการ
ผลผลิตเดือนนี้ 186 กก.
```

#### 6.2.3 เมนูด่วน

มี 4 ปุ่ม

| ปุ่ม | Action |
|---|---|
| บันทึกการปลูก | ไปหน้าบันทึกการปลูก |
| รายการปลูก | ไปหน้ารายการปลูก |
| บันทึกเก็บเกี่ยว | ไปหน้าบันทึกเก็บเกี่ยว |
| สรุปผลผลิต | ไปหน้าสรุป หรือส่วนสรุปผลผลิต |

#### 6.2.4 ใกล้ถึงวันเก็บเกี่ยว

แสดงรายการที่ยังไม่เก็บเกี่ยว และเรียงตามวันเก็บเกี่ยวจากใกล้ที่สุด

ข้อมูลที่แสดงต่อรายการ:

```txt
ชื่อผัก
วันที่ปลูก
วันที่เก็บเกี่ยวได้
เหลืออีกกี่วัน
```

ตัวอย่าง:

```txt
Green Cos
ปลูก 05/06/2026 • เก็บเกี่ยว 30/06/2026
เหลือ 2 วัน
```

#### 6.2.5 ปลูกล่าสุด

แสดงรายการปลูกล่าสุด 2-3 รายการ

ข้อมูลที่แสดง:

```txt
ชื่อผัก
วันที่ปลูก
วันที่คาดว่าเก็บเกี่ยว
```

### 6.3 พฤติกรรมของหน้าแรก

- เมื่อเปิดเว็บ ให้โหลดข้อมูลจาก Google Apps Script
- ถ้าโหลดไม่สำเร็จ ให้แสดงข้อความแจ้งเตือนแบบสุภาพ
- ถ้ายังไม่มีข้อมูล ให้แสดง empty state เช่น `ยังไม่มีรายการปลูก`
- รายการใกล้เก็บเกี่ยวควรแสดงไม่เกิน 3-5 รายการ เพื่อไม่ให้หน้าแน่น

## 7. หน้าบันทึกการปลูก

### 7.1 วัตถุประสงค์ของหน้า

ใช้สำหรับบันทึกรายการปลูกใหม่ และให้ระบบคำนวณวันที่สามารถเก็บเกี่ยวได้โดยอัตโนมัติ

### 7.2 องค์ประกอบที่ต้องมี

#### 7.2.1 Header

```txt
GrowDay
บันทึกการปลูก
```

#### 7.2.2 ข้อความอธิบายสั้น

```txt
กรอกข้อมูลการปลูกเพื่อคำนวณวันเก็บเกี่ยวอัตโนมัติ
```

#### 7.2.3 ฟอร์มบันทึกการปลูก

| Field | Type | Required | รายละเอียด |
|---|---|---|---|
| ชนิดผัก | dropdown | Yes | เลือกจากรายการผัก 10 ชนิด |
| หน่วยปลูก | dropdown | Yes | ค่าเริ่มต้นคือ จำนวนช่องปลูก |
| จำนวนที่ปลูก | number | Yes | จำนวนช่องปลูกหรือจำนวนต้น |
| วันที่ปลูก | date | Yes | วันที่ย้ายต้นกล้าเข้าสู่รางปลูก |
| คาดว่าเก็บเกี่ยวได้ | read-only card | Auto | คำนวณจากวันที่ปลูก + อายุเก็บเกี่ยว |
| หมายเหตุ | textarea | No | ระบุรายละเอียดเพิ่มเติม |

### 7.3 รายชื่อผักใน dropdown

```txt
Green Cos
Mini Cos
Green Oak
Red Oak
Finlay
Green Coral
Red Coral
Butterhead
Kale
Rocket
```

### 7.4 หน่วยปลูก

ค่าเริ่มต้นที่แนะนำ:

```txt
จำนวนช่องปลูก
```

ตัวเลือกอื่นที่สามารถมีได้:

```txt
จำนวนต้น
จำนวนถาด
จำนวนราง
```

แต่ใน MVP ให้ใช้ `จำนวนช่องปลูก` เป็นหลัก เพราะเหมาะกับระบบปลูกผักไฮโดรโปนิกส์ และสัมพันธ์กับการวางแผนผลผลิตได้ง่าย

### 7.5 Logic การคำนวณวันเก็บเกี่ยว

เมื่อเลือกชนิดผักและวันที่ปลูก ระบบต้องคำนวณวันเก็บเกี่ยวอัตโนมัติ

```txt
วันที่เก็บเกี่ยว = วันที่ปลูก + อายุเก็บเกี่ยวของผักชนิดนั้น
```

ตัวอย่าง:

```txt
Green Cos
วันที่ปลูก: 05/06/2026
อายุเก็บเกี่ยว: 25 วัน
วันที่เก็บเกี่ยว: 30/06/2026
```

### 7.6 ปุ่มในหน้า

| ปุ่ม | Action |
|---|---|
| บันทึกการปลูก | ตรวจข้อมูลและบันทึกลง Google Sheets |
| ล้างข้อมูล | เคลียร์ข้อมูลในฟอร์ม |

### 7.7 Validation

ก่อนบันทึกต้องตรวจสอบ

- ต้องเลือกชนิดผัก
- ต้องระบุจำนวนที่ปลูก
- จำนวนที่ปลูกต้องมากกว่า 0
- ต้องระบุวันที่ปลูก
- วันที่ปลูกต้องเป็นวันที่ถูกต้อง
- หากหมายเหตุยาวเกิน 200 ตัวอักษร ให้แจ้งเตือนหรือจำกัดความยาว

### 7.8 หลังบันทึกสำเร็จ

หลังจากกดบันทึกสำเร็จ ให้ทำอย่างใดอย่างหนึ่ง

ทางเลือกที่แนะนำ:

```txt
แสดงข้อความ "บันทึกการปลูกเรียบร้อยแล้ว" แล้วล้างฟอร์ม
```

หรือ

```txt
แสดงข้อความสำเร็จ แล้วพาผู้ใช้ไปหน้า "รายการปลูก"
```

สำหรับ MVP แนะนำให้แสดงข้อความสำเร็จและล้างฟอร์ม เพื่อให้ผู้ใช้บันทึกหลายรายการต่อเนื่องได้ง่าย

## 8. หน้ารายการปลูก

### 8.1 วัตถุประสงค์ของหน้า

ใช้ดูรายการปลูกทั้งหมดที่บันทึกไว้ และจัดการรายการ เช่น แก้ไข หรือบันทึกเก็บเกี่ยว

### 8.2 องค์ประกอบที่ต้องมี

#### 8.2.1 Header

```txt
GrowDay
จัดการการปลูกผักไฮโดรโปนิกส์
```

#### 8.2.2 ชื่อหน้า

```txt
รายการปลูก
```

#### 8.2.3 Search

ช่องค้นหาชื่อผัก

```txt
ค้นหาชื่อผัก
```

#### 8.2.4 Filter Chips

| Filter | ความหมาย |
|---|---|
| ทั้งหมด | แสดงทุกรายการ |
| กำลังปลูก | แสดงรายการที่ยังไม่ถึงวันเก็บเกี่ยว |
| พร้อมเก็บ | แสดงรายการที่ถึงวันเก็บเกี่ยวแล้วแต่ยังไม่บันทึกผลผลิต |
| เก็บแล้ว | แสดงรายการที่บันทึกผลผลิตแล้ว |

### 8.3 Summary ย่อยของหน้ารายการปลูก

แสดงการ์ดเล็ก 2 ใบ

```txt
ทั้งหมด 24 รายการ
พร้อมเก็บ 3 รายการ
```

### 8.4 รูปแบบรายการปลูก

แม้ในเอกสารสเปกจะเรียกว่า “ตาราง” แต่บนมือถือให้ใช้รูปแบบ Card List แทนตาราง เพื่อให้อ่านง่าย

ข้อมูลที่แสดงต่อหนึ่งรายการ:

| ข้อมูล | ตัวอย่าง |
|---|---|
| ชื่อผัก | Green Cos |
| วันที่ปลูก | 05/06/2026 |
| วันที่เก็บเกี่ยวได้ | 30/06/2026 |
| สถานะ / อีกกี่วัน | เหลือ 2 วัน / พร้อมเก็บ / เก็บแล้ว |
| ผลผลิต | 12.5 กก. หรือ - |
| ปุ่มจัดการ | แก้ไข / เก็บเกี่ยว |

ตัวอย่าง Card:

```txt
Green Cos
วันที่ปลูก 05/06/2026
เก็บเกี่ยวได้ 30/06/2026
เหลือ 2 วัน
ผลผลิต 12.5 กก.
[แก้ไข] [เก็บเกี่ยว]
```

### 8.5 สถานะของรายการ

| Status | เงื่อนไข | Label |
|---|---|---|
| growing | ยังไม่ถึงวันเก็บเกี่ยว | กำลังปลูก / เหลือ X วัน |
| ready | ถึงวันเก็บเกี่ยวแล้ว แต่ยังไม่บันทึกผลผลิต | พร้อมเก็บ |
| harvested | บันทึกผลผลิตแล้ว | เก็บแล้ว |
| cancelled | ยกเลิกรายการ | ยกเลิก |

### 8.6 ปุ่มในแต่ละรายการ

#### ปุ่มแก้ไข

ใช้แก้ไขข้อมูลรายการปลูก เช่น

- ชนิดผัก
- จำนวนที่ปลูก
- วันที่ปลูก
- หมายเหตุ

ใน MVP สามารถทำแบบง่ายได้ เช่น เปิด modal หรือโหลดข้อมูลกลับไปที่ฟอร์มเดิม

#### ปุ่มเก็บเกี่ยว

ใช้ไปยังหน้าบันทึกเก็บเกี่ยว พร้อมส่ง id ของรายการปลูกไปด้วย

หากรายการมี status = harvested แล้ว ไม่ต้องแสดงปุ่มเก็บเกี่ยว

### 8.7 Empty State

ถ้ายังไม่มีรายการปลูก ให้แสดงข้อความ

```txt
ยังไม่มีรายการปลูก
เริ่มจากการบันทึกการปลูกครั้งแรก
[บันทึกการปลูก]
```

## 9. หน้าบันทึกเก็บเกี่ยว

### 9.1 วัตถุประสงค์ของหน้า

ใช้บันทึกผลผลิตจริงหลังจากเก็บเกี่ยวผักแล้ว

### 9.2 องค์ประกอบที่ต้องมี

#### 9.2.1 Header

```txt
GrowDay
บันทึกเก็บเกี่ยว
```

#### 9.2.2 ข้อความอธิบายสั้น

```txt
เลือกแปลงหรือรายการปลูกที่ถึงรอบเก็บเกี่ยว แล้วบันทึกผลผลิตจริง
```

#### 9.2.3 ฟอร์มบันทึกเก็บเกี่ยว

| Field | Type | Required | รายละเอียด |
|---|---|---|---|
| เลือกรายการปลูก | dropdown | Yes | แสดงเฉพาะรายการที่ยังไม่ harvested |
| ข้อมูลรายการปลูก | read-only info box | Auto | วันที่ปลูก, กำหนดเก็บเกี่ยว, จำนวนช่องปลูก |
| วันที่เก็บเกี่ยวจริง | date | Yes | ค่าเริ่มต้นเป็นวันนี้ |
| ปริมาณผลผลิต | number | Yes | เช่น 12.5 |
| หน่วย | dropdown | Yes | ค่าเริ่มต้น กก. |
| หมายเหตุ | textarea | No | รายละเอียดเพิ่มเติม |

### 9.3 ข้อมูลใน dropdown รายการปลูก

ตัวอย่าง:

```txt
Green Cos • ปลูก 05/06/2026 • พร้อมเก็บ
Red Oak • ปลูก 07/06/2026 • พร้อมเก็บ
Mini Cos • ปลูก 06/06/2026 • เหลือ 1 วัน
```

หมายเหตุ: ใน MVP แนะนำให้แสดงเฉพาะรายการที่ `status = ready` ก่อน เพื่อป้องกันการบันทึกผิดรายการ หากต้องการให้บันทึกก่อนกำหนดได้ ค่อยเพิ่มในรุ่นถัดไป

### 9.4 Info Box รายการปลูก

หลังเลือก dropdown ให้แสดงข้อมูลสรุป

```txt
วันที่ปลูก: 05/06/2026
กำหนดเก็บเกี่ยว: 30/06/2026
จำนวนช่องปลูก: 120
```

### 9.5 ปุ่มในหน้า

| ปุ่ม | Action |
|---|---|
| บันทึกผลผลิต | บันทึกวันที่เก็บเกี่ยวจริงและผลผลิตจริง |

### 9.6 Validation

ก่อนบันทึกต้องตรวจสอบ

- ต้องเลือกรายการปลูก
- ต้องระบุวันที่เก็บเกี่ยวจริง
- ต้องระบุปริมาณผลผลิต
- ปริมาณผลผลิตต้องมากกว่า 0
- ต้องระบุหน่วยผลผลิต

### 9.7 หลังบันทึกสำเร็จ

หลังจากบันทึกสำเร็จ

- เปลี่ยน status ของรายการเป็น `harvested`
- บันทึก actualHarvestDate
- บันทึก actualYield
- บันทึก yieldUnit
- แสดงข้อความ `บันทึกผลผลิตเรียบร้อยแล้ว`
- อัปเดตรายการเก็บเกี่ยวล่าสุด

### 9.8 เก็บเกี่ยวล่าสุด

แสดงรายการที่เก็บเกี่ยวล่าสุด 3 รายการ

ข้อมูลที่แสดง:

```txt
ชื่อผัก
วันที่เก็บเกี่ยวจริง
ปริมาณผลผลิต
```

ตัวอย่าง:

```txt
Red Oak
29/06/2026
10.8 กก.
```

## 13. การแสดงวันที่

### 13.1 รูปแบบแสดงผลบนหน้าเว็บ

ให้แสดงวันที่เป็นรูปแบบอ่านง่ายสำหรับผู้ใช้ไทย

```txt
DD/MM/YYYY
```

ตัวอย่าง:

```txt
05/06/2026
30/06/2026
```

### 13.2 รูปแบบจัดเก็บในระบบ

สำหรับ Google Sheets และ Apps Script แนะนำให้จัดเก็บเป็น

```txt
YYYY-MM-DD
```

ตัวอย่าง:

```txt
2026-06-05
2026-06-30
```

### 13.3 กติกาสำคัญ

- ห้ามคำนวณวันที่จาก string ที่ไม่แน่นอน
- ให้แปลงวันที่ให้เป็น Date object ก่อนคำนวณ
- ระวัง timezone ของ Apps Script
- ควรใช้รูปแบบ `yyyy-MM-dd` ในการส่งข้อมูลระหว่าง frontend กับ backend

## 20. Acceptance Criteria ด้าน UI/UX

ถือว่างาน UI Flow ผ่านเมื่อ

- เปิดเว็บแล้วเห็นหน้าแรกโดยไม่ต้อง Login
- หน้าแรกแสดงสรุปรายการปลูกและผลผลิตได้
- ผู้ใช้สามารถไปหน้าบันทึกการปลูกได้จากหน้าแรกและ Bottom Nav
- ฟอร์มบันทึกการปลูกคำนวณวันที่เก็บเกี่ยวอัตโนมัติ
- ผู้ใช้สามารถดูรายการปลูกทั้งหมดได้ในรูปแบบที่อ่านง่ายบนมือถือ
- ผู้ใช้สามารถกรองรายการปลูกตามสถานะได้
- ผู้ใช้สามารถบันทึกผลผลิตจริงได้
- หลังบันทึกเก็บเกี่ยว สถานะเปลี่ยนเป็น `เก็บแล้ว`
- Bottom Navigation แสดงสถานะเมนูที่กำลังใช้งานถูกต้อง
- หน้าเว็บใช้งานบนมือถือได้โดยไม่ต้องเลื่อนซ้ายขวา
- UI ไม่ซับซ้อนเกินกว่าที่สร้างได้จริงด้วย Tailwind CDN + Vanilla JS

## 22. สรุป

`UI_FLOW.md` เป็นเอกสารควบคุมหน้าจอและลำดับการใช้งานของ GrowDay รุ่น MVP โดยเน้นให้ระบบใช้งานง่ายบนมือถือ สร้างได้จริงด้วยเทคโนโลยีที่กำหนด และไม่ขยายขอบเขตเกินจำเป็น

หน้าจอที่สำคัญที่สุดใน MVP คือ

```txt
หน้าแรก → บันทึกการปลูก → รายการปลูก → บันทึกเก็บเกี่ยว → สรุปผลผลิตพื้นฐาน
```

หากต้องแก้ไข UI ในอนาคต ควรแก้เอกสารนี้ก่อน แล้วจึงให้ Codex หรือทีมพัฒนาปรับโค้ดตามเอกสาร
